## Executive Summary

This is a single-operator anonymous pastoral chat — architecturally small, but with three sharp edges: identity that must survive months with no account, reachability that must survive iOS, and machine translation that must never author a word. The closest prior art is anonymous helpline chat, not sales live-chat (whose feature set is mostly actively wrong here). Everything the researchers converged on follows from that reframing.

The recommended build is the locked stack with two corrections: **Next.js 16.2.10 + React 19.2.7 + TypeScript pinned to 6.0.3** (TS 7 is Go-native, dropped `lib/typescript.js`, breaks Next 16.2 TS detection), **Postgres 17 + Drizzle 0.45.2 on postgres.js 3.4.9** (whose `sql.listen()` gives a dedicated non-pooled connection with reconnect and an `onlisten` replay hook for free), `jose` sessions, hand-written `public/sw.js`, `web-push` 3.6.7, OpenAI SDK → OVHcloud. **Contradiction with PROJECT.md: bcrypt must be replaced by `@node-rs/argon2`** — OWASP now classes bcrypt as legacy-only, and it is a node-gyp native module that fails with `invalid ELF header` in a glibc-builder → musl-Alpine-runner build. *(Corrected — see orchestrator note.)*

Risks are concentrated and knowable. Translation is a chat completion, so it is prompt-injectable in both directions. And Swahili is unresolved on every candidate OVH model, gating the language list, which gates the locale files.

## Convergent Findings (highest confidence)

1. **The anonymous ID must be a server-set `HttpOnly` `Secure` `SameSite=Lax` cookie, not a client-generated `localStorage` UUID.** A client-supplied ID is a bearer token readable by any XSS on the origin; and WebKit ITP deletes all script-writable storage after 7 days without interaction — destroying exactly the slow-burn conversations this product exists for. `localStorage` keeps language/appearance plus a recovery copy; cookie wins on conflict. Add the push endpoint as a third identity anchor (`endpoint → visitor_id`), plus a signed ID in the push click URL and in the PWA `start_url`.
2. **Push payloads must be content-free** — a localized "You have a new reply" + conversation id. In several target locales a lock-screen preview of a faith conversation is a physical-safety risk, not just a privacy one.
3. **The OVH translation spike moves INTO Phase 1, before the 10 locale JSON files are written.** No candidate model documents Swahili; the answer changes the language list, the input to every localized string, the picker and the RTL work. Extend the spike to a prompt-injection test set and the heuristic validators.
4. **`LISTEN/NOTIFY` is fire-and-forget and cannot alone satisfy "no message is ever lost."** Cursor replay — `id: <messages.id>` on every event, `Last-Event-ID` on connect, `WHERE conversation_id=$1 AND id > $2 ORDER BY id` — is **Phase 1 architecture, not Phase 3 hardening**. NOTIFY is a latency optimization on a polling-correct design. Backfill *is* history loading; no separate history endpoint.
5. **The connection budget must be fixed, not O(visitors).** (a) `LISTEN` on a pooled client is per-connection session state — the pool reassigns/reaps the socket and realtime dies *silently* hours after deploy. (b) An SSE handler holding a DB client for the stream's life exhausts a 10-connection pool at ~10 visitors and blocks every write. Correct: **one dedicated listener + one pool (max ~10) = 11 connections forever**, in-process `Map<convId, Set<Subscriber>>` fan-out, streams connection-free while idle.

## Key Findings

**Stack.** Node 24.18 LTS on `node:24-alpine` (Node 26 is Current — avoid). `typescript@6.0.3` pinned exactly. postgres.js for both pool and listener (removes ~100 lines of hand-rolled reconnect). `@node-rs/argon2` 2.0.2 over bcrypt (prebuilt musl binaries); fallback `bcryptjs@3.0.3`, never `bcrypt@6.0.0`. Dockerfile build pack, not Nixpacks — `output: 'standalone'` does not trace `.next/static`, `public/`, or `drizzle/`; migrations run at container start. Coolify: app and DB land on different Docker networks by default (enable *Connect To Predefined Network*); `NEXT_PUBLIC_VAPID_PUBLIC_KEY` must be build-time, `DATABASE_URL`/`VAPID_PRIVATE_KEY`/`OVH_API_KEY`/`SESSION_SECRET` runtime-only. SSE: `export const dynamic = 'force-dynamic'` is load-bearing (otherwise ISR buffers the whole response); disable compression on the route, `no-transform` + `X-Accel-Buffering: no`, 20s heartbeat, expect Traefik long-connection flakiness (coolify#4002, #8298) and design for deliberate ~4-min reconnects. **VAPID key loss is the only unrecoverable event in the system** — one-time generation, backed up off-box, never generated in a Dockerfile or startup script; rotation permanently unreachable-ifies every existing visitor.

**Features.** *Table stakes:* full-screen bubble thread with restored history; optimistic send with a *single* delivery state; explicit "a real person answers this"; auto-detected/overridable/persisted language and appearance; full RTL; two-way translation with the original always retrievable; localized push on reply; mobile-usable owner inbox; rate limiting; block + delete. *Differentiators:* the URL *is* the chat; exactly two header controls; per-message collapsed "show original" on both sides (a safeguard, since no faith glossary ships in v1); owner-side translation preview before send; manual faith-decision flag driving a pastoral sort; honest presence replacing the offline email form; guided iOS install; EU (Gravelines) inference residency. *Anti-features:* AI/auto-reply or suggested replies (one AI reply retroactively poisons trust in every real one), canned responses, typing indicators (dots that start and stop read as "he gave up on me"), read receipts, CSAT, offline email capture, pre-chat forms, queue positions, auto-close, automated moderation, automatic faith-decision detection, analytics dashboards, multi-agent routing UI, email/SMS push fallback, file upload, message editing, transcript export, in-page sounds. *Defer:* inbox filters/search, glossary, second provider, re-subscription recovery, second responder (schema-ready, zero UI), voice notes.

**Architecture.** Seven tables: `responders` (seeded with one row — owner language and presence live here, 80% of the multi-responder seam), `visitors` (server-generated uuid, `ip_hash` HMAC never raw IP, `push_lost_at`), `conversations` (denormalized `last_*_at`; partial unique index enforcing one open conversation per visitor at DB level), `messages` (`body` = the author's original, **immutable**; `client_msg_id` for free idempotency; `id` doubles as the SSE event id), `message_translations` (**separate table** — `(message_id, target_lang)` PK *is* the cache, carries queue state, survives an owner language change and a second responder; `skipped` for same-language pairs is the highest-leverage cost optimization), `push_subscriptions` (unique on `endpoint`, hard-deleted on 404/410), `rate_limit_buckets` (DB-backed, survives frequent redeploys). **Translation is asymmetric by design:** visitor→owner async *after* durability (an LLM endpoint must never be a prerequisite for persistence); owner→visitor sync against the *draft* before a row exists, then persisted `status='ready'` in the same transaction — the reply arrives already translated on the first SSE frame at zero extra OVH cost. **Degradation contract:** `messages.body` is `NOT NULL` and is the render fallback in every state; never an empty bubble or a spinner in place of the message. **Push fires from the write path gated by a durable `delivered_at` ACK + ~8s grace**, not an in-memory "is an SSE open?" check (wrong across tab-freeze, restart, replicas); the SW must **always** `showNotification()` — browsers penalize silent pushes and can revoke permission, which here means losing the visitor. Stamp `assigned_responder_id` on first reply from day one even though nothing reads it.

**Pitfalls (top items).**

| # | Pitfall | Prevention |
|---|---------|------------|
| 1 | iOS push gate turns away visitors | `requestPermission()` as the *first statement* in the click handler — any `await` launders the gesture. Guided Add-to-Home-Screen screen for iOS Safari; ID carried through `start_url` (the tab and the installed PWA are separate storage partitions) |
| 2 | Anonymous ID evaporates (ITP 7-day, install partition split, private mode) | Signed HttpOnly cookie as source of truth + localStorage recovery copy + push-endpoint recovery + signed ID in the push click URL |
| 3 | Subscriptions rot silently; owner replies into the void | Reap on 404/410, flag the conversation **unreachable in the dashboard**, re-sync `getSubscription()` on every open (Chrome does not fire `pushsubscriptionchange`), round-trip probe on grant, `push_outbox` + retry worker |
| 4 | Translator answers the visitor instead of translating | Structural message separation + JSON-shaped output + validators (script block, length ratio, refusal markers, emoji/URL/digit preservation) → **fail toward the original**; never feed translated text into any automated decision. Crisis and abuse disclosures are the messages most likely to trigger a model refusal — exactly the ones that must never go invisible |
| 5 | Bidi mangling (URLs, Bible references, digits, mirrored logos) | Two direction axes (`<html dir>` = UI locale, per-message `dir` = message language), `<bdi>` isolation, `text-align: match-parent`, mirror by allowlist never blanket `scaleX(-1)`, mixed-direction fixture set as a **Phase 1** deliverable |
| 6 | SSE + LISTEN/NOTIFY silently stops delivering | See Convergent Findings 4 and 5 |
| 7 | Coolify loses the conversations | Named volume verified by a deliberate restart test; off-box `pg_dump` **enabled in Phase 1**; one documented restore drill; `log_statement=none` + parameterized writes so confessions never reach the web-accessible log viewer |
| 8 | One human, a stranger in crisis, no escalation — by design | Expectation-setting welcome copy (Phase 1 — it is a locale string); honest presence; localized crisis resources; away mode; admin lockout + revocable sessions + TOTP; never over-promise "safe/private" beyond what the architecture delivers |

## Implications for Roadmap

Architecture and pitfalls independently rejected the naive split (chat → push+translation → dashboard) for the same two reasons: the spike lands too late, and the realtime spine goes unexercised until Phase 3 because there is no reply surface to test it with.

**Phase 1 — Foundation and the realtime spine.** *Rationale:* every later phase rides on the message pipeline, and two Phase-1 outputs (the language list, the identity model) are inputs to work in the same phase. *Delivers:* OVH spike (Swahili + injection set) → migrations for all seven tables → pool + dedicated listener + hub + SSE with `Last-Event-ID` backfill → server-set cookie identity → send endpoint + token-bucket limiter → locale JSON + RTL + mixed-direction fixture set → `<Gate>` shell bypassed by env flag + SW registration + manifest → **minimal owner login and reply surface** → presence read path → Postgres volume + off-box backups. *Pitfalls:* 2, 5, 6, 7, 8 (expectation copy).

**Phase 2 — Reachability and language.** *Rationale:* both external dependencies isolated to one phase, on a pipeline already proven. *Delivers:* real push gate, localized decline re-ask, iOS A2HS screen with an animated Share-sheet visual → `push_subscriptions` + send + 404/410 reaping + `delivered_at` ACK + push worker + round-trip probe → translate worker with backoff, circuit breaker, `(message_id, target_lang)` cache → owner draft preview → "show original". *Pitfalls:* 1, 3, 4. **Exit criterion: real-iPhone testing across Safari tab and installed PWA — not a Phase 3 task.**

**Phase 3 — Owner surface, hardening, ship.** *Rationale:* purely additive UI over a schema that already carries every field. *Delivers:* inbox priority sort / filters / search / counts → status controls + faith-decision flag → presence toggle UI → block + delete → per-locale crisis resources → admin lockout + revocable sessions + TOTP → icon-mirroring audit → health check, Traefik SSE verification, VAPID generation + off-box backup, restore drill. *Pitfalls:* 3, 5, 7, 8.

**Research flags.** *Needs research:* Phase 2 — iOS push under a real installed PWA (backgrounded SW wake, >7-day storage durability, subscription rot) is the weakest-confidence area in the corpus and cannot be simulated; plus OVH translation quality/injection resistance, which only the spike answers. *Standard patterns, skip:* Phase 3 (inbox UI, filters, search, auth hardening). Phase 1 is well-documented except Traefik long-lived-SSE behaviour, which should be verified empirically at first deploy rather than researched.
