<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Service worker → notification display | Runs in a privileged browser context with no server round trip for the display step itself. |
| Client-side platform/standalone detection | Purely advisory UI branching; never a security boundary (both branches lead to the same backend-enforced flow). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-22 | Denial of Service (self-inflicted, permission revocation) | sw.js push handler | high | mitigate | `showNotification()` is unconditional on every push event — no silent-push code path exists that could trigger a browser's permission-revocation heuristic (Pitfall 4). |
| T-02-23 | Tampering | notificationclick URL construction | medium | mitigate | The click-handler's `vid` value is read only from the notification's own `data` field (server-constructed at send time, never client-editable after the fact) — a compromised page script cannot alter an already-delivered notification's data payload. |
| T-02-24 | Information Disclosure | icon placeholder generation script | low | accept | A hand-rolled PNG encoder has a narrow, well-understood output (a single solid color, no external input) — no injection surface exists in a script with zero external/user input. |

</threat_model>
