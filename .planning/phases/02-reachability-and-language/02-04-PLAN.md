<objective>
Build the push subscribe/probe backend, the ACK-aware, 404/410-cleanup-aware push sender, the gate-funnel beacon endpoint, and the client-side helpers Plan 02-07's Gate UI will call: subscribe-to-push, the every-open re-sync check, and the beacon sender.

Output: `src/server/push/{subscribe,send,gateEvent}.ts`, their thin route wrappers, and `src/lib/push/subscribe-client.ts`.

Per **D-13**, `send.ts`'s `sendNotification` call uses web-push's `topic` field (`conv-${conversationId}`) so multiple unread replies while the visitor's phone is locked coalesce into one notification, never stacking as separate individual notifications for the same conversation.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Visitor browser → /api/push/subscribe, /api/push/gate-event | Unauthenticated-by-identity (visitor-cookie-scoped, but the endpoint/subscription object itself is fully client-supplied and untrusted). |
| App → push service (browser vendor's push endpoint) | Outbound network call carrying the content-free notification payload. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-09 | Spoofing | /api/push/subscribe | high | mitigate | `visitorId` for the `pushSubscriptions` insert is always derived from the authenticated `requireVisitor()` session cookie server-side, never trusted from the request body — mirrors `send.ts`'s existing pattern of never trusting a client-supplied identifier. |
| T-02-10 | Tampering | gate-event beacon replay | medium | mitigate | `gateFunnel`'s COALESCE-based, set-once-only upsert (Plan 02-01) caps a replayed beacon's effect to a no-op after the first call — no separate anti-replay token needed. |
| T-02-11 | Denial of Service | /api/push/subscribe, /api/push/gate-event flooding | medium | mitigate | Both routes are visitor-session-scoped (one visitor identity per caller); a future hardening pass can add `ratelimit.ts`'s existing token-bucket if abuse is observed — not added here since Phase 1's OPS-01 already rate-limits the message-send path an attacker would need to reach first to obtain a visitor session at all. |
| T-02-12 | Information Disclosure | push payload construction | high | mitigate | Payload is built exclusively from fixed locale strings + a routing token — verified by the region-scoped negative grep in Task 1/2 that no message-body reference exists in either file. |
| T-02-13 | Tampering | subscription object validation | medium | mitigate | zod schema requires well-formed `{endpoint, keys:{p256dh,auth}}` before any DB write; a malformed object 400s before ever reaching `pushSubscriptions.create`. |

</threat_model>
