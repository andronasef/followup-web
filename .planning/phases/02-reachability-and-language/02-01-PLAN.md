<objective>
Lay the shared backend substrate both verticals this phase build on: the three schema additions (a new `push_gate_funnel` table, a uniqueness guarantee on `message_translations`, and a `delivered_at` ACK column on `messages`), the `web-push` dependency, the VAPID config seam, and the push/funnel repo layer. Also add two small, targeted repo/config additions (`OWNER_LANG`, `getVisitorLangFor`) that the translation vertical needs later.

Purpose: every other plan in this phase (push subscribe/send, translation wiring, identity recovery, and both UI waves) imports from the modules this plan creates. Building and migrating the schema first, in its own plan, means every later plan works against a live, already-migrated local database instead of discovering a missing column mid-task.
Output: a migrated schema, `web-push` installed, `src/server/push/vapid.ts`, `src/server/repo/pushSubscriptions.ts`, `src/server/repo/gateFunnel.ts`, and small additions to `messages.ts`/`models.ts`/`conversations.ts`.

Per **D-01**, the owner generates the production VAPID keypair themselves, off-box, via `npx web-push generate-vapid-keys` on their own machine — this plan never generates or handles a real production private key, only the config seam that reads it from env. Per **D-02**, `VAPID_PRIVATE_KEY` is documented (in `.env.example`'s comment and this plan's `user_setup`) as backed up by the owner in a second location independent of Dokploy. Per **D-03**, the keypair is treated as permanent; `vapid.ts`'s fail-loud-on-missing-env design exists precisely because rotation is a break-glass action, never a routine one — a silently-missing key must never be mistaken for "just restart and it'll regenerate."
</objective>
