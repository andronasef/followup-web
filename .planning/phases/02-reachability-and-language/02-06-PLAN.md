<assumption_delta_decision>
## Assumption-Delta: a second visitor-identity anchor (ID-03)

**Signal:** this phase adds a push-subscription-endpoint-based identity anchor alongside the existing cookie (primary) + localStorage-mirror (backstop) pair from Phase 1 — a pluralization of "how a visitor is recognized."

**Decision: add-alongside, not promote.** The signed `HttpOnly` cookie remains the sole primary identity anchor everywhere it is present. `localStorage` remains a display/recovery mirror only (Phase 1's D-11, unchanged). The push-endpoint recovery (ID-03) and the URL-carried vid-token (ID-04) are BOTH strictly best-effort, recovery-only anchors, consulted only in the narrow case where the primary anchor is absent for this storage context — they never override an existing, valid cookie, and neither is ever read as a source of truth for an already-identified request.

**Rationale for add-alongside over promote:** promoting either anchor to primary would mean trusting a client-supplied endpoint string or URL parameter as identity ahead of a signed, `HttpOnly` cookie — a strictly worse security posture for zero UX gain in the common case (cookie present). The ONLY case these anchors exist for is the cookie's own storage context being absent (iOS's separate Home-Screen-app storage jar, or a rare full storage wipe) — a genuinely new second/third case, not a replacement for the first.

**Accepted debt:** a race exists where `pre-paint.ts`'s existing no-cookie bootstrap path may mint a brand-new (eventually-orphaned) visitor+conversation row a few hundred milliseconds before an ID-03 recovery check (client-side, requires an async service-worker round trip) can complete and correct the cookie back to the recovered identity. This is accepted as low-risk, harmless waste (one extra empty conversation row, never visitor-facing) rather than re-architecting `pre-paint.ts`'s synchronous, pre-first-paint bootstrap to block on an async recovery check — which would reintroduce the exact "flash of wrong content" problem `pre-paint.ts` exists to avoid. ID-04's iOS path does NOT have this race (see Task 3): the vid-token is checked synchronously in the SAME bootstrap call, before any visitor row would be minted.
</assumption_delta_decision>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client-supplied vid token / recover endpoint → visitor identity | Both are recovery-only anchors that must never let one visitor assume another's identity. |
| Visitor client → /api/chat/messages/ack | Fully client-controlled messageId input. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-18 | Spoofing | /api/push/recover, vid-token flow | high | mitigate | Both mechanisms only ever resolve to an identity that was already cryptographically established (a `jose`-signed JWT verified against the app's own secret, or a `push_subscriptions` row keyed by an endpoint the server itself issued that subscription against) — never a client-supplied raw visitor id. A forged/tampered vid token fails `verifySession` and falls through to the existing mint-new-visitor path, same as an invalid cookie. |
| T-02-19 | Spoofing | /api/chat/messages/ack | high | mitigate | `messages.belongsToConversation` is checked before any write — a visitor can only ack a message inside their OWN session-derived conversation id, never an arbitrary message id from another conversation (T-02-19 directly addresses the risk of one visitor suppressing push to another). |
| T-02-20 | Repudiation / Tampering | admin push-send trigger | medium | mitigate | The push trigger is derived entirely from the already-authenticated `requireOwner()` reply flow's own persisted result (conversationId/messageId/visitorId), never from a second, independently-trusted input. |
| T-02-21 | Information Disclosure | vid-token endpoint | medium | mitigate | `GET /api/visitor/vid-token` is `requireVisitor()`-guarded and only ever returns a token for the CALLER's own already-established session — it cannot be used to mint or read another visitor's token. |

</threat_model>
