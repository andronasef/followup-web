<objective>
Wire Phase 1's proven, durability-first message-send transactions into the live translation pipeline: async visitor→owner translation (fire-and-forget after persistence), synchronous owner→visitor draft-preview (before send, editable), and the shared cache/validation layer both paths use. Extend the shared `since`/`sinceAll` message query so every "translated + original" pair automatically flows through the existing SSE/polling/SSR fetch paths with zero changes to the route files themselves.

Purpose: this is the vertical slice that makes "the owner reads Swahili in English, and can reply in English shown as Swahili" real end-to-end at the data/API layer, before Plan 02-08 builds the actual toggle/composer UI on top of it. TRANS-05's durability-first guarantee (already Phase 1's hardest-won property) must not move — translation is added alongside the existing transaction boundary, never inside it.
Output: `messageTranslations.ts`, `cache.ts`, the extended `messages.ts`/`send.ts`/`route.ts`/`reply.ts`, and the new `translate-preview` endpoint.

Per **D-09**, the owner-facing draft-preview is an inline swap (the composer's own textbox is what carries draft, then translated, text) — this plan's `translate-preview.ts` endpoint and `reply.ts`'s `originalBody` field exist specifically to support that inline-swap UX (built in Plan 02-08), not a side-by-side comparison view. Per **D-11**, `reply.ts`'s existing behavior (sending whatever text is currently in the composer, translated or not) already IS the untranslated-original fallback when translation fails/times out — reaffirmed, not renegotiated, by this plan's design.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Visitor message body → NVIDIA NIM (async, after persistence) | Untrusted visitor text crosses into the translation call, but only after CHAT-06's durability guarantee already holds. |
| Owner draft text → NVIDIA NIM (sync, pre-send) | Owner-authored text (still untrusted in the sense that it's free text) crosses into the preview call before any persistence. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-14 | Tampering | translation-cache write ordering | high | mitigate | `after()` is called only from `route.ts` (never `send.ts`), preserving CHAT-06/TRANS-05's durability-first guarantee — a translation failure or slow provider can never delay or block the visitor's 200 response. |
| T-02-15 | Information Disclosure | translate-preview.ts error path | medium | mitigate | A failed/timed-out translate-preview call returns a generic `{failed:true}` shape, never echoing raw provider error text (which could itself contain the untranslated draft) back in a way that could reach logs — matches OPS-09's existing no-message-bodies-in-logs discipline. |
| T-02-16 | Elevation of Privilege (of the LLM's instruction-following) | translate-preview.ts, cache.ts | high | mitigate | Both call sites go through the SAME structurally-isolated `translate()` function (Plan 02-02) — no bespoke prompt construction is introduced here that could reopen the TRANS-08 injection surface. |
| T-02-17 | Tampering | reply.ts's originalBody field | low | accept | A malicious/compromised owner session could POST an arbitrary `originalBody`; the owner is the single authenticated operator of this system (ADMIN-01) and this field only ever affects that owner's own "see original" display, never a security-relevant value — accepted as in-scope-of-trust, matching Phase 1's existing owner-trust boundary. |

</threat_model>
